## 1. 需求确认摘要

### 1.1 最终确认需求

| 序号 | 需求项 | 确认结果 |
|------|--------|----------|
| Q1 | 行业选择方式 | **预设卡片式选项**（政府/电商/短视频/网约车/货运/即时配送/自定义） |
| Q2 | 场景类型 | **系统自动推断**（根据行业自动匹配场景规则） |

| Q9 | 元数据内容 | 行业、场景、文件路径、生成时间、数量 |
| Q10 | 并发限制 | **单次限制**，前端防抖 + 后端去重 |

## 2. 架构审视报告

### 2.1 需求文档问题分析

以需求准入架构师视角，对现有需求文档进行审视，发现以下问题：

#### 🔴 严重问题（必须解决）

| 序号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| P-01 | **EvalTool 评估指标未定义** | 无法量化生成质量，验收标准缺失 | 需定义量化评分标准（准确率、召回率、格式正确率） |
| P-02 | **流量生成的数据格式未明确** | 前后端联调困难 | 需定义 TrafficData 数据模型 |
| P-03 | **外部 API 接口规范缺失** | TrafficIdentityTool 无法实现 | 需明确外部系统接口协议 |

#### 🟡 中等问题（建议解决）

| 序号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| P-04 | 缺少安全需求定义 | 无法评估安全风险 | 补充认证、授权、数据加密需求 |
| P-05 | 监控可观测性设计缺失 | 生产环境运维困难 | 补充日志、指标、追踪设计 |
| P-06 | 部署架构设计不足 | 无法评估生产部署方案 | 补充容器化、K8s 部署方案 |
| P-07 | 测试策略不完整 | 质量保障不足 | 补充单元/集成/E2E 测试策略 |

#### 🟢 优化建议（可选）

| 序号 | 问题 | 建议 |
|------|------|------|
| O-01 | 项目规范缺失 | 补充代码规范、Git Flow |
| O-02 | 性能基线未定义 | 定义 P99 延迟、吞吐量指标 |
| O-03 | 数据备份策略缺失 | 补充备份恢复方案 |

### 2.2 架构可行性评估

| 评估项 | 结论 | 说明 |
|--------|------|------|
| **技术可行性** | ✅ 通过 | LangChain+LangGraph 组合成熟，Ollama 本地部署稳定 |
| **业务可行性** | ⚠️ 待确认 | 需明确业务边界和验收标准 |

## 3. 业务需求规格

### 3.1 业务背景

构建一个基于 **LangChain + LangGraph + FastAPI** 的结构化任务导向型 Agent 系统，用于智能交通场景下的**流量生成任务**。

### 3.2 核心业务流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                         流量生成主流程                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌──────────┐    ┌─────────────┐    ┌────────────┐    ┌─────────┐│
│  │ 用户输入  │───▶│ RAG 检索    │───▶│ 流量生成   │───▶│ 质量评估 ││
│  │ 行业/场景 │    │ (Milvus)    │    │ (LLM)      │    │ (Eval)  ││
│  └──────────┘    └─────────────┘    └────────────┘    └────┬────┘│
│                                                              │      │
│                                                    ┌─────────▼─────┐│
│                                                    │  评估通过?    ││
│                                                    └───────┬───────┘│
│                                                            │        │
│                                          ┌─────────────────┘        │
│                                          ▼                          ▼
│                              ┌───────────────┐            ┌───────────────┐
│                              │  身份化判断   │            │   重试(≤3次)  │
│                              │ (Identity)   │            └───────────────┘
│                              └───────┬───────┘
│                                      ▼
│                              ┌───────────────┐
│                              │  输出带身份标签│
│                              │  的模拟流量   │
│                              └───────────────┘
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 功能需求矩阵

| 功能模块 | 功能点 | 优先级 | 验收标准 | 状态 |
|----------|--------|--------|----------|------|
| **F1: RAG 检索** | F1.1 行业案例检索 | P0 | 基于行业/场景检索，返回 Top-5 相关案例 | 待开发 |
| | F1.2 相似度阈值过滤 | P1 | 相似度 < 0.6 时触发降级 | 待开发 |
| **F2: 流量生成** | F2.1 基于案例生成流量 | P0 | 生成格式符合 TrafficData 标准 | 待开发 |
| | F2.2 生成数量控制 | P1 | 支持指定生成数量 1-100 | 待开发 |
| **F3: 质量评估** | F3.1 格式正确性评估 | P0 | 返回格式评分 0-100 | 待开发 |
| | F3.2 业务相关性评估 | P0 | 返回业务评分 0-100 | 待开发 |
| | F3.3 评估阈值判定 | P0 | 总分 ≥ 70 分视为通过 | 待开发 |
| **F4: 身份化判断** | F4.1 车辆身份验证 | P0 | 返回身份标签（真实/模拟/异常） | 待开发 |
| | F4.2 规则引擎打分 | P1 | 支持可配置的规则打分 | 待开发 |
| **F5: 异常处理** | F5.1 向量库延期 | P0 | Milvus初始化暂缓，预留接口与降级策略 | 待开发 |
| | F5.2 评估与超时重试 | P0 | 超时/评分不及格最多重试3次，若仍不满意返回最新结果 | 待开发 |
| | F5.3 身份校验宕机处理 | P1 | 外部 API/规则引擎宕机时请求失败（严禁 LLM 兜底） | 待开发 |

### 3.4 数据输入/输出规格

#### 2.4.1 输入规格

```python
class TrafficGenerationRequest(BaseModel):
    industry: str = Field(..., description="行业类型，如 '网约车'/'货运'/'物流'")
    scenario: str = Field(..., description="场景类型，如 '早高峰'/'晚高峰'/'节假日'")
    count: int = Field(default=10, ge=1, le=100, description="生成数量")
    enable_identity_check: bool = Field(default=True, description="是否启用身份校验")
    metadata: Optional[dict] = Field(default=None, description="额外元数据")
```

#### 2.4.2 输出规格

```python
class TrafficData(BaseModel):
    id: str = Field(..., description="流量唯一标识 (UUID)")
    plate_number: str = Field(..., description="车牌号")
    vehicle_type: str = Field(..., description="车辆类型")
    timestamp: datetime = Field(..., description="时间戳")
    location: Location = Field(..., description="位置信息")
    speed: float = Field(..., ge=0, description="速度 (km/h)")
    direction: str = Field(..., description="行驶方向")
    identity_label: Optional[str] = Field(None, description="身份标签: real/fake/anomaly")

class TrafficGenerationResponse(BaseModel):
    success: bool
    total_count: int
    generated_data: List[TrafficData]
    quality_score: QualityScore
    processing_time_ms: int
    session_id: str
```

### 4.3 状态定义

```python
class TrafficAgentState(TypedDict):
    messages: Annotated[list, operator.add]  # 对话历史
    current_node: str                         # 当前节点
    retry_count: int                          # 重试计数
    max_retries: int                          # 最大重试次数

    # 上下文数据
    input_request: TrafficGenerationRequest   # 输入请求
    retrieved_cases: List[dict]               # RAG 检索结果
    generated_traffic: List[TrafficData]       # 生成的流量
    quality_scores: QualityScore               # 质量评分
    identity_results: List[IdentityResult]    # 身份判断结果

    # 执行状态
    execution_trace: List[ExecutionStep]      # 执行轨迹
    error_info: Optional[ErrorInfo]           # 错误信息

    # 输出结果
    final_result: Optional[TrafficGenerationResponse]
```

### 4.4 组件职责

| 组件 | 职责 | 边界 |
|------|------|------|
| **Supervisor Agent** | 协调整个流程，决定下一步执行 | 编排调度，不执行业务逻辑 |
| **RAG Node** | 检索行业案例，调用 MilvusRAGTool | 仅负责检索，降级策略内置 |
| **Generate Node** | 调用 LLM 生成流量 | 调用 Ollama，返回 TrafficData[] |
| **Eval Node** | 评估生成质量 | 调用 EvalTool，判定是否通过 |
| **Identity Node** | 车辆身份校验 | 调用 TrafficIdentityTool |

```python
class MilvusRAGToolInput(BaseModel):
    query: str = Field(..., description="检索查询字符串")
    top_k: int = Field(default=5, ge=1, le=20, description="返回结果数量")
    similarity_threshold: float = Field(default=0.6, ge=0, le=1)

class MilvusRAGToolOutput(BaseModel):
    cases: List[TrafficCase] = Field(..., description="检索到的案例")
    fallback_used: bool = Field(default=False, description="是否使用降级策略")
    search_time_ms: int

class MilvusRAGTool(BaseTool):
    name = "milvus_rag_search"
    description = "从向量知识库检索行业流量案例"
    args_schema = Type[MilvusRAGToolInput]

    def _run(self, query: str, top_k: int = 5, similarity_threshold: float = 0.6) -> str:
        try:
            results = self.milvus_client.search(
                collection_name="traffic_cases",
                query_text=query,
                top_k=top_k
            )
            filtered = [r for r in results if r.score >= similarity_threshold]
            if not filtered:
                return self._fallback(query)
            return json.dumps(filtered, ensure_ascii=False)
        except ConnectionError:
            logger.warning("Milvus connection failed, using fallback")
            return self._fallback(query)

    def _fallback(self, query: str) -> str:
        """降级策略：返回通用行业模板"""
        return json.dumps([{"type": "generic_template", "content": query}], ensure_ascii=False)
```

### 6.7 限流与并发控制（QPS设计）
为保护本地硬件环境，尤其是 GPU 显存限制（如 RTX 2060 6GB 极易溢出），系统对请求并发做如下设计：
- **现阶段：全局排他锁（单并发）**。通过 SQLite 或 FastAPI 内存锁机制保证同一时刻只有一个 Agent 任务在执行。若有新请求进入，直接返回 `HTTP 429 Too Many Requests`，提示用户“系统正忙，请稍后重试”。
- **架构预留：令牌桶（Token Bucket）限流**。代码需在路由中间件或依赖注入层实现 `RateLimiter` 接口，以便后期硬件升级时可通过配置文件放开 QPS（如 5 QPS），而不必重构业务代码。

### 6.8 Milvus 向量库与初始化数据（阶段性延期）
- **设计承诺**：架构上完整保留 `MilvusRAGTool` 的能力定义与调用节点。相关检索参数和数据结构不受影响。
- **落地现状**：由于当前未就绪初始化行业案例数据，且考虑到本地显存运行 Embedding 模型和 LLM 可能会存在资源竞争。
- **阶段方案**：一期开发中，RAG节点默认返回配置好的 Mock 样例或降级为直连生成阶段，真正的数据库拉起和数据初始化加入到二期开发 TODO List。
