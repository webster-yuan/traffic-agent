## 3. 业务需求规格

### 3.1 业务背景

构建一个基于 **LangGraph Supervisor-Worker 多智能体架构 + LangChain + FastAPI** 的结构化任务导向型 Agent 系统，用于智能交通场景下的**流量生成任务**。以 LLM 为决策中心的 Supervisor 动态调度 RAG 检索、流量生成（嵌套子图）、质量评估（Pandera 声明式校验）、人工审批（HITL interrupt）、身份校验 5 个专业化 Worker，实现高真实度行业场景模拟流量的自动化合成与全链路治理。

**v3.1 新增能力**：
- **Token 消耗追踪**：`token_counter.py` 从 Ollama 响应提取 `prompt_eval_count` / `eval_count` / `total_duration_ns`，通过 SSE `token_usage` 事件实时推送每次 LLM 调用的 Token 明细，前端生成完成后展示 Token 统计面板（调用次数 / 输入 Tokens / 输出 Tokens / 总 Tokens / 总耗时 / 平均生成速度）
- **模型信息展示**：`GET /api/v1/traffic/model-info` 端点扩展返回 `model_name` / `provider` / `context_window` / `stages`，前端页面标题旁显示模型徽章（hover 展示 Provider / Context Window）
- **批量失败重试**：`POST /batch/{batch_id}/retry-failed` 一键重试失败任务，`BatchPanel.vue` 自动检测失败任务并显示「🔄 重试失败任务」按钮
- **中文编码修复**：`supervisor.py`/`workers.py` 的 SSE 阶段名和思想链消息 UTF-8 乱码修复；`trafficStore.ts` `STAGE_NAME` 补充 `approval: '人工审核'` 映射

### 3.2 核心业务流程（实际实现）

```
┌───────────────────────────────────────────────────────────────────────┐
│                     Supervisor-Worker 多智能体流程                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  START                                                                  │
│    │                                                                     │
│    ▼                                                                     │
│  ┌─────────────────┐                                                    │
│  │   Supervisor    │  LLM (ChatOllama) 分析 state                        │
│  │   决策中心       │  输出 Pydantic RouterDecision                       │
│  └────────┬────────┘  ──→ "rag" / "generate" / "eval" / "approval" / "identity"      │
│           │                                                              │
│     ┌─────┼─────────────────┬──────────────┬───────────────┐           │
│     ▼     ▼                 ▼              ▼               ▼           │
│  ┌────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐ ┌────────────┐     │
│  │RAG │ │ Generate │ │   Eval   │ │  Approval  │ │  Identity  │     │
│  │    │ │  Worker  │ │  Worker  │ │   Worker   │ │   Worker   │     │
│  │    │ │  ┌─────────────────┐ │ │            │ │            │     │
│  │    │ │  │ 子图 (Subgraph) │ │ │  HITL      │ │  real/fake/│     │
│  │    │ │  │ prompt→LLM→parse│ │ │  interrupt │ │  anomaly   │     │
│  │    │ │  └─────────────────┘ │ │            │ │            │     │
│  └──┬─┘ └────┬─────┘ └────┬─────┘ └─────┬──────┘ └─────┬──────┘     │
│     │        │            │              │              │            │
│     └────────┴────────────┴──────────────┴──────────────┘            │
│                     │                                                  │
│          Command(goto="supervisor") ──→ 循环回 Supervisor              │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────┐          │
│  │ 并行扇出 (Send API) — full 模式:                          │          │
│  │   Send("eval", state) + Send("identity", state) 并发执行  │          │
│  └──────────────────────────────────────────────────────────┘          │
│                                                                         │
│    Supervisor 再次决策 → END (FINISH) 或 retry (generate)               │
└───────────────────────────────────────────────────────────────────────┘
```

**核心特点**（与 v1.0 线性流程对比）:
- v1.0: 固定 RAG→Gen→Eval→Identity 顺序执行
- v2.0: Supervisor 在每步后**重新决策**，支持重试/跳过/并行/审批等动态路由
- v2.0: full 模式下 eval 和 identity **并行扇出**，节省等待时间
- v2.0: Generate Worker 内部封装**独立子图**，可单独测试
- v2.1: **Human-in-the-Loop** — Approval Worker 通过 `interrupt()` 暂停图等待人工审批
- v2.2: **Prompt 自优化** — eval 失败反馈注入 generate subgraph 实现自我校正

    approval_action: str                # "approve" | "reject" | ""

## 6. 详细设计 — 核心实现模式

### 6.1 Supervisor 结构化路由决策

Supervisor 是 LangGraph 多智能体的核心。它不执行具体任务，而是分析全局 state 后决定下一个 Worker。

```python
# graph/supervisor.py

_SYSTEM_PROMPT = """你是 Traffic Agent 系统的 Supervisor（主控代理）。
你的职责是根据当前系统状态，决定下一个应该执行的 Worker。

可用 Worker：
- rag      — 检索行业流量案例和推断场景（首次必执行）
- generate — 调用 LLM 生成流量记录
- eval     — 对生成的流量进行质量评分（格式/业务/多样性）
- identity — 身份标签校验（全量模式）
- FINISH   — 所有任务完成，结束流程

决策规则：
1. retrieved_cases 为空 → rag
2. rag 完成，generated_records 为空 → generate
3. generate 完成 → eval
4. eval 后：
   - quality_passed=true → identity（full）或 FINISH
   - should_retry=true → generate
   - should_retry=false 且 quality_passed=false → FINISH
5. identity_checked=true 或 stage != full → FINISH
6. error_message 非空 → FINISH

返回 JSON: {"next": "<worker>", "reason": "<决策原因>"}"""

async def supervisor_node(state: GraphState) -> Command[Any]:
    # --- 并行派发检查 (P2.3) ---
    stage = state.get("stage")
    if (stage == Stage.full and state.get("generated_records")
            and not state.get("identity_checked")
            and state.get("quality_score", QualityScore()).total_score == 0):
        return Command(goto="__parallel__", update={"next_worker": "__parallel__"})

    # --- LLM 结构化路由 ---
    llm = ChatOllama(model=settings.ollama_model, temperature=0.1, timeout=300)
    structured_llm = llm.with_structured_output(RouterDecision, method="json_mode")
    
    try:
        decision: RouterDecision = await structured_llm.ainvoke(messages)
    except Exception:
        decision = _fallback_route(state)  # 降级到纯规则引擎
    
    return Command(
        goto=decision.next,
        update={"next_worker": decision.next, "messages": [AIMessage(...)]}
    )
```

**设计要点**:
- `with_structured_output(RouterDecision, method="json_mode")` 强制 Ollama 输出合法 RouterDecision 对象
- LLM 失败时有 `_fallback_route()` 纯规则降级，保证可用性
- `reason` 字段被提取到前端 SSE 事件中，用户可见决策过程

### 6.2 Send() 并行扇出实现

LangGraph 的 `Send()` 必须从 **conditional edge** 路由函数返回，不能在 node 内部直接 return `list[Send]`。

```python
# graph/supervisor.py

def route_supervisor(state: GraphState) -> str | list[Send]:
    """Conditional-edge 路由函数 — 支持 Send() 返回 List[Send]"""
    next_worker: str = state.get("next_worker", "")
    
    if next_worker == "__parallel__":
        return [
            Send("eval", _make_send_state(state, "eval")),
            Send("identity", _make_send_state(state, "identity")),
        ]
    return next_worker  # 普通字符串路由

def _make_send_state(state: GraphState, worker: str) -> dict[str, Any]:
    """构建 Send() 所需的最小 state 子集"""
    return {
        "session_id": state.get("session_id"),
        "industry": state.get("industry"),
        "stage": state.get("stage"),
        "generated_records": state.get("generated_records", []),
        # ... 其他 Worker 需要的字段
        "next_worker": worker,
    }

# graph/workflow.py — 注册路由
builder.add_conditional_edges(
    "supervisor",
    route_supervisor,
    {
        "rag": "rag", "generate": "generate",
        "eval": "eval", "identity": "identity",
        "__end__": END, "FINISH": END,
        "__parallel__": END,  # fallback, Send() 已处理真正派发
    },
)
```

**设计要点**:
- `Send()` 返回类型是 `list[Send]`，与 `str` 路由共存于同一个函数
- `Send()` 创建的是孤立子任务，各自有独立 state
- 并行 Worker 执行完毕后都回到 Supervisor，合并状态

### 6.3 Generate 子图嵌套

Generate Worker 内部封装独立 StateGraph，便于独立测试和复用。

```python
# graph/generate_subgraph.py

class GenerateSubState(TypedDict):
    industry: str; scenario: str; count: int; stage: Stage
    prompt: str; raw_response: str; records: list[TrafficRecord]; error: str

def build_generate_subgraph() -> StateGraph:
    builder = StateGraph(GenerateSubState)
    builder.add_node("prepare_prompt", prepare_prompt_node)
    builder.add_node("call_llm", call_llm_node)
    builder.add_node("parse_result", parse_result_node)
    builder.add_edge(START, "prepare_prompt")
    builder.add_edge("prepare_prompt", "call_llm")
    builder.add_edge("call_llm", "parse_result")
    builder.add_edge("parse_result", END)
    return builder.compile()

# graph/workers.py — 父图调用子图
async def generate_worker(state: GraphState) -> Command[dict]:
    subgraph = build_generate_subgraph()
    sub_state = {
        "industry": state["industry"], "scenario": state["scenario"],
        "count": state["count"], "stage": state["stage"],
        "prompt": "", "raw_response": "", "records": [], "error": "",
    }
    result = await subgraph.ainvoke(sub_state)
    return Command(
        goto="supervisor",
        update={"generated_records": result.get("records", [])}
    )
```

**设计要点**:
- 子图有独立的 `GenerateSubState`，与父图 `GraphState` 类型隔离
- 子图内部 3 个节点在 `astream_events` 中有独立 namespace
- 子图可单独 run：`subgraph.ainvoke({...})` 测试

### 6.5 SSE Agent 思考链流式 (Custom Streaming)

利用 LangGraph `astream` v2 API 的 `stream_mode=["updates","custom"]` 捕获节点输出和自定义事件，转为 SSE 推送前端。

```python
# api/routes.py

async def event_stream() -> AsyncGenerator[str, None]:
    graph = get_traffic_graph()
    async for (mode, data) in graph.astream(
        initial_state, config=graph_config,
        stream_mode=["updates", "custom"],  # updates: 节点输出, custom: get_stream_writer()
    ):
        # ── Custom 事件 (来自节点内的 get_stream_writer()) ──
        if mode == "custom":
            evt_type = data.get("type", "")
            
            if evt_type == "stage_start":
                # 阶段开始: {"node": "rag", "name": "RAG 检索", ...}
                yield f"event: stage_start\ndata: {json.dumps(data)}\n\n"
            
            elif evt_type == "thought":
                # Supervisor 思考过程
                yield f"event: thought\ndata: {json.dumps(data)}\n\n"
            
            elif evt_type == "generate_progress":
                # 生成进度: {"type": "generate_progress", "current": 5, "total": 10}
                yield f"event: generate_progress\ndata: {json.dumps(data)}\n\n"
            continue
        
        # ── updates 事件 — 节点输出 ──
        for node_name, state_update in data.items():
            # HITL interrupt 检测
            if node_name == "__interrupt__":
                yield f"event: waiting_for_approval\ndata: {json.dumps(...)}\n\n"
                return
            
            # Supervisor 决策提取
            if node_name == "supervisor":
                yield f"event: thought_decision\ndata: {json.dumps({'decision': ...})}\n\n"
            
            # 阶段完成
            if node_name in stage_name_map:
                yield f"event: stage_complete\ndata: {json.dumps(...)}\n\n"
```

**踩坑记录**:
- LangGraph `astream()` 压制 `GraphInterrupt`，以 `updates` 事件 `{"__interrupt__": ...}` 形式抛出 → SSE handler 需检测 `__interrupt__` 并发射 `waiting_for_approval`
- Supervisor 决策通过 `state_update` 中的 `messages` 或 `next_worker` 提取，不依赖 `on_chain_end`
- `stream_mode="custom"` 的事件来自节点内 `get_stream_writer()` 调用，需注意 `RuntimeError`（非 streaming 上下文时）

### 6.6 降级容错：Fallback 路由

```python
def _fallback_route(state: GraphState) -> RouterDecision:
    """纯 Python 规则引擎 — 零 LLM 依赖，100% 可靠"""
    retrieved_count = len(state.get("retrieved_cases", []) or [])
    generated_count = len(state.get("generated_records", []) or [])
    quality_score = state.get("quality_score")
    
    if state.get("error_message"):
        return RouterDecision(next="FINISH", reason="检测到错误，终止")
    if retrieved_count == 0:
        return RouterDecision(next="rag", reason="尚未检索案例")
    if generated_count == 0:
        return RouterDecision(next="generate", reason="开始生成流量")
    if quality_score is None or quality_score.total_score == 0:
        return RouterDecision(next="eval", reason="开始质量评估")
    if state.get("should_retry"):
        return RouterDecision(next="generate", reason="质量未达标，重新生成")
    if not state.get("quality_passed"):
        return RouterDecision(next="FINISH", reason="已达最大重试次数")
    if state.get("stage") == Stage.full and not state.get("identity_checked"):
        return RouterDecision(next="identity", reason="全量模式身份校验")
    return RouterDecision(next="FINISH", reason="所有步骤完成")
```

### 7.3 任务终止机制

```
用户点击取消 → DELETE /api/v1/traffic/generate/{session_id}
                    ↓
            memory 中设置 is_cancelled(session_id) = True
                    ↓
            每个 Worker 节点入口 _check_cancelled(session_id) 检查
                    ↓
            检测到取消 → raise CancelledError
                    ↓
            清理临时文件 → 返回终止成功
```

**已知限制**: LLM 调用期间（`llm.ainvoke()`）无法检查取消标记，需等 LLM 返回后才识别。

## 附录 A: Agent 架构核心亮点

### A.1 Supervisor-Worker vs 传统 Linear Chain

- 传统: RAG→Gen→Eval→Identity 串行（固定 DAG）
- 本项目: Supervisor 每步**重新决策**，支持重试/跳过/并行
- LLM 驱动但降级有保障: `with_structured_output()` + `_fallback_route()`

### A.2 Send() 并行扇出

- `route_supervisor()` 返回 `str | list[Send]` 双类型
- full 模式: `[Send("eval",...), Send("identity",...)]` 并发
- 并行 Worker 独立 state → 执行完回到 Supervisor 合并

### A.3 子图嵌套

- `GraphState` vs `GenerateSubState` 类型隔离
- 子图 3 节点 (prompt → LLM → parse) 独立可测
- `astream_events` 独立 namespace 细粒度展示

### A.4 SSE Agent 思考链

- `on_chat_model_start/stream/end` → LLM 推理追踪
- `on_chain_end` → Supervisor RouterDecision 提取
- Pinia `thoughts[]` + Vue `v-for` 滚动渲染

### A.5 可靠性三层保障

1. 结构化输出失败 → `_fallback_route()` 纯规则降级
2. 质量不通过 → Supervisor 自动重试路由 (max_retries=3) + eval_feedback 自优化
3. Checkpoint 持久化 → `AsyncSqliteSaver` 每步可回溯，`aget_state_history()` Time Travel 回放

### A.6 HITL 人工审批

- LangGraph `interrupt()` 在 Approval Worker 中动态暂停图
- SSE `waiting_for_approval` 事件推送审批面板（统计摘要 + 样例记录）
- `POST /resume` + `Command(resume=...)` 注入 Approve/Reject 决策恢复执行
- Reject → 自动回到 Generate 重生成 → 再次审批，形成闭环
