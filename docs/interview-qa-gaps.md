# Traffic Agent — 技术面试潜在问题与能力缺口分析

> **定位**：从技术面试官视角，梳理项目已具备 vs 缺失的能力，聚焦 Agent 服务与流量生成核心领域。
> **排除**：环境/Docker/K8s/Grafana/硬件相关（假定模型部署在最新 GPU 上）。
> **状态标注**：✅ 已具备 | 🔧 可快速实现 | 📋 需设计说明 | 🔮 远期规划

---

## 一、性能与压测

### Q1: "这个系统做过压测吗？最大并发、吞吐量数据是多少？"

**现状**：
- ✅ `asyncio.Semaphore(3)` 并发控制
- ✅ 批量最多 10 任务
- ❌ 无任何压测脚本/工具
- ❌ 无吞吐量（records/sec）统计
- ❌ 无 QPS/TPS 数据

**缺口**：缺少系统化的性能基准数据。面试官会追问"3并发是基于什么定的？有没有测过极限？"

**🔧 可实施**：
- 添加 `backend/tests/test_benchmark.py` 压测脚本，覆盖单会话/并发5/10/最大场景
- 在 `/api/v1/traffic/metrics` 端点暴露实时吞吐量

**📋 需设计说明**（详见第六章）：
- 分布式压测框架设计（Locust/k6）
- 不同模型（7B/14B/72B）下的性能对比矩阵

---

### Q2: "单条流量生成的端到端延迟分布是怎样的？P50/P95/P99？"

**现状**：
- ✅ `processing_time_ms` 返回总耗时
- ❌ 无分段耗时统计（RAG/生成/评估各自耗时）
- ❌ 无延迟分位数统计
- ❌ 无历史延迟趋势

**缺口**：面试官想知道你对系统性能瓶颈的理解深度。当前只有一个笼统的 `processing_time_ms`。

**🔧 可实施**：
- 在 SSE 流中增加各阶段耗时（已有 `elapsed_ms`，但未聚合）
- `/api/v1/traffic/metrics` 返回 P50/P95/P99 延迟分布
- 内存滑动窗口统计最近 100 次请求的延迟分位数

---

### Q3: "LLM 调用是核心瓶颈，Token 消耗效率如何？每条记录消耗多少 Token？"

**现状**：
- ❌ 完全不追踪 Token 消耗
- ❌ 无输入/输出 Token 计数
- ❌ 无成本估算

**缺口**：这是 AI Agent 系统的核心指标。没有 Token 追踪 = 无法做成本优化。

**🔧 可实施**：
- 接入 Ollama API 的 `eval_count` / `prompt_eval_count` 字段
- 添加 `token_counter.py` 服务，统计每次调用的 Token
- `/api/v1/traffic/metrics` 返回 token 效率（tokens/record）

---

## 二、流量生成效率

### Q4: "一次 LLM 调用生成所有记录，大批量（500+条）时上下文窗口够吗？怎么保证不截断？"

**现状**：
- ✅ `count` 上限 10000（Pydantic 校验）
- ❌ 一次调用生成全部，无分批策略
- ❌ 无上下文长度检查
- ❌ LLM 返回被截断时仅靠 JSON 修复

**缺口**：qwen2.5:7b 上下文窗口 32K tokens。500+ 条记录 + examples + system prompt 很容易超出。面试官会问"你测试过最大生成量吗？"

**📋 需设计说明**：
- 分批生成 + 结果拼接策略
- 上下文窗口自适应检测
- Token 预算分配模型（examples 占多少、prompt 占多少、输出占多少）

---

### Q5: "流量生成的实时性如何？从请求到第一笔数据返回要多久？"

**现状**：
- ✅ SSE 流式推送各阶段进度
- ❌ 无首字节时间（TTFB）统计
- ❌ 生成过程中用户看不到部分结果

**缺口**：流式生成是 AI 产品的核心体验指标。当前 SSE 只推进度事件，不推部分生成结果。

**📋 需设计说明**：
- Streaming partial results 方案（逐条推送已生成的记录）
- TTFB 优化策略（RAG 缓存 + prompt 预热）

---

## 三、质量保障深度

### Q6: "Pandera 15条规则之外，有没有自动化回归测试来保证生成质量不退化？"

**现状**：
- ✅ Pandera 格式 13 字段 + 业务 6 跨字段规则
- ✅ 三维度评分（格式 30% + 业务 40% + 多样性 30%）
- ❌ 无 golden dataset（标注好的"正确答案"）
- ❌ Prompt 改了之后，没有回归对比
- ❌ 无质量趋势追踪

**缺口**：Prompt 工程师最怕的就是"改了一个 prompt，不知道其他行业会不会退化"。

**📋 需设计说明**：
- Golden Dataset 构建方案（每个行业 5-10 条人工标注的标杆流量）
- 自动化回归评估流水线（改 prompt → 对 12 行业跑一遍 → 对比质量分）
- 质量退化告警阈值

---

### Q7: "质量评分只有 70 分的二元通过线，有没有更细粒度的质量等级？"

**现状**：
- ✅ `passed = total_score >= 70`
- ❌ 无分层（如 90+ 优秀、70-89 合格、<70 不合格）
- ❌ 不合格时的降级策略单一（重试 3 次 → 放弃）

**缺口**：生产环境中"70分刚好过"和"95分优秀"的流量数据，下游使用价值完全不同。

**🔧 可实施**：
- 添加质量等级枚举（excellent/good/acceptable/poor）
- 在 API 响应和前端报表中展示质量等级

---

## 四、Agent 可靠性

### Q8: "Supervisor 的 LLM 路由失败了怎么办？整个流程就断了吗？"

**现状**：
- ✅ `_fallback_route()` 确定性降级路由
- ✅ `max_retries=3` 重试
- ❌ 降级路由不够智能（只有简单的状态机判断）
- ❌ 无部分成功处理（比如生成5条中3条通过质量检查）

**缺口**：这就是 Agent 系统的"鲁棒性"问题。面试官可能追问"如果 eval worker 挂了但 generate 成功了，能直接返回不评估的结果吗？"

**📋 需设计说明**：
- 多级降级策略（LLM路由失败 → 规则路由 → 跳过可选 Worker → best-effort 返回）
- Circuit breaker 模式（连续失败 N 次后暂停某 Worker）
- Partial success 返回机制

---

### Q9: "重试3次全失败后，用户看到什么？有没有兜底方案？"

**现状**：
- ✅ 3 次重试后 `should_retry=false` → FINISH
- ❌ 无兜底生成（如用规则模板生成代替 LLM）
- ❌ 无失败原因分类（JSON解析失败 vs 质量不达标 vs LLM超时）

**缺口**：用户体验很差——等了 3 轮 LLM 调用，最后得到一个空结果。

**🔧 可实施**：
- 失败时回退到 `generate_records()` 规则模板生成
- 在 API 响应中区分失败类型

---

### Q10: "批量生成中，10个任务有2个失败，用户拿到的是全部失败还是8个成功的？"

**现状**：
- ✅ 批量创建 + 轮询状态
- ❌ 失败任务不会重试
- ❌ 用户无法只下载成功部分
- ❌ 批量结果无聚合下载

**缺口**：Partial success 是批量系统的常见需求。

**🔧 可实施**：
- 批量状态增加 `partial_success` 状态
- 批量结果聚合下载（所有成功任务的 CSV 合并）
- 失败任务一键重试端点

---

## 五、可观测性

### Q11: "除了 LangSmith，系统自身有哪些监控和告警？"

**现状**：
- ✅ LangSmith `@traceable` 追踪
- ✅ SSE 思考链推送
- ❌ 无自定义 metrics（Prometheus 格式）
- ❌ 无健康度 dashboard
- ❌ 无异常告警

**缺口**：LangSmith 是外部服务。面试官会问"如果 LangSmith 挂了，你怎么排查问题？"

**🔧 可实施**：
- `/api/v1/traffic/metrics` 端点（Prometheus 兼容格式）
- 内存中维护滑动窗口指标（请求总数、成功率、平均延迟、Token消耗）
- 前端增加简单的 health dashboard

---

### Q12: "Token 消耗能按行业/场景/用户维度统计吗？"

**现状**：
- ❌ 无 Token 计数基础设施
- ❌ 无按维度聚合的 Token 报表

**🔧 可实施**：
- SQLite 新增 `token_usage` 表
- 每次 LLM 调用记录 token 消耗
- 历史 API 支持按 token 排序/过滤

---

## 六、流量真实性

### Q13: "生成流量的统计分布和真实流量一致性如何验证？有没有引入时序模式？"

**现状**：
- ✅ identity_label 75%/25% real/fake 分布
- ✅ 随机化方法/状态码/IP
- ❌ 无真实流量基线对比
- ❌ 无统计假设检验（K-S test、卡方检验）
- ❌ 时间戳均匀分布，无业务高峰/低谷
- ❌ 无周期性模式

**缺口**：网络安全产品需要高仿真流量做训练。面试官会问"你怎么证明生成的流量和真实流量在统计上不可区分？"

**📋 需设计说明**：
- 统计一致性验证框架（K-S 检验、分布相似度量化）
- 时序模式注入（按小时的业务高峰、按季节的流量变化）
- 行业特征差异化（电商双11流量模式 vs 政务工作日模式）

---

### Q14: "怎么保证不同行业的流量特征差异化？有没有行业专属的生成策略？"

**现状**：
- ✅ 12 行业 RAG 示例
- ✅ 行业 URL 路径
- ✅ `IndustryConfig` 声明式模板
- ❌ 示例只有 3 条/行业，覆盖不够
- ❌ 无行业专属质量检查（如金融行业的敏感字段掩码）

**📋 需设计说明**：
- RAG 示例动态扩充机制（成功案例自动入库）
- 行业专属 Pandera 规则（金融脱敏、医疗合规）

---

## 七、安全性

### Q15: "API 有没有鉴权？如何防止恶意调用消耗 Token？"

**现状**：
- ✅ `slowapi` 速率限制（200/day）
- ✅ CORS 限制 origin
- ❌ 无 API Key / JWT 鉴权
- ❌ 无用户维度隔离
- ❌ 无 Token 配额管理

**缺口**：生产环境中，没有鉴权 = 任何人都能调用，Token 消耗无法控制。

**📋 需设计说明**：
- JWT 鉴权方案设计
- 用户级 Token 配额
- 输入注入防护（prompt injection 检测）

---

## 八、扩展性

### Q16: "支持切换不同模型吗？有没有做过模型对比评估？"

**现状**：
- ❌ 单模型 `qwen2.5:7b`
- ❌ 无模型切换机制
- ❌ 无 A/B 对比框架

**📋 需设计说明**：
- 多模型配置与动态切换
- 同一场景多模型生成 → 质量对比报告

---

### Q17: "Prompt 怎么管理？有没有版本控制和回滚机制？"

**现状**：
- ❌ Prompt 硬编码在代码中
- ❌ 无版本管理
- ❌ 改 prompt 需要改代码发版

**📋 需设计说明**：
- Prompt 模板外部化（YAML/JSON 配置文件）
- Git-like prompt 版本控制
- 灰度发布（10% 流量用新 prompt）
