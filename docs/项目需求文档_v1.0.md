## 1. 项目概述

### 1.1 项目名称
**Traffic Agent** - 智能交通 Agent 系统

### 1.2 项目目标
构建一个基于 **LangChain + LangGraph + FastAPI** 的结构化任务导向型 Agent 系统，用于智能交通场景下的复杂任务处理。

### 1.3 核心价值
- 多 Agent 协作处理复杂交通任务
- RAG 增强的上下文理解
- 可持久化的状态管理
- 本地化部署，无外部依赖
- **流式输出** - 实时反馈处理进度

### 1.4 技术优先级
| 优先级 | 技术 | 说明 |
|--------|------|------|
| 🔴 核心 | LangGraph StateGraph + Supervisor | **必须掌握** - 框架核心 |
| 🔴 核心 | SQLite 持久化 | **必须掌握** - 状态管理 |
| 🟡 重要 | Ollama 本地 LLM | **必须掌握** - 模型集成 |
| 🟡 重要 | 多 Agent 协作模式 | **必须掌握** - 架构设计 |
| 🟢 辅助 | Milvus Lite RAG | ⭐ 中等权重 - 用于学习 RAG 模式落地 |

## 4. Agent 核心特征

本系统是基于 **LangChain+FastAPI** 落地的结构化任务型智能体（Structured Task-oriented Agent），核心匹配 Agent 的五大特征。

### 4.1.1 目标驱动（Goal-Oriented）

Agent 核心是"给定目标自主完成闭环"。当前工具输入"行业/场景"，通过 FastAPI 接口触发后，无需人工干预，自动完成全流程：

```
输入：行业/场景参数
    ↓
RAG检索案例 → LLM生成流量 → 质量评估 → 身份化判断
    ↓
输出：带身份标签的行业模拟流量
```

所有步骤均围绕"生成符合业务特征的高质量流量"这一核心目标推进，是典型的任务型 Agent 目标驱动特征。

### 4.1.2 工具调用能力（Tool Use）

Agent 的核心能力是调用外部工具扩展自身能力边界，而非仅依赖 LLM 原生能力：

| 工具 | 类型 | 能力扩展 |
|------|------|----------|
| **MilvusRAGTool** | 检索工具 | 调用 Milvus 向量库获取行业场景的真实流量案例，解决 LLM "无行业数据"的问题 |
| **EvalTool** | 质量评估 | 流量生成后的质量校验（EVAL + 重试机制） |
| **TrafficIdentityTool** | 身份判断 | 调用规则引擎 + LLM 完成流量身份打分，解决"流量真实性校验"问题 |

工具均遵循 LangChain 的 BaseTool 标准封装，可被 Chain/Agent 统一调度。

### 4.1.3 决策与规划（Decision-Making & Planning）

Agent 的"规划能力"体现为"按目标拆解执行步骤"。当前基于 SequentialChain 实现固定流程：

```
第一步规划：先检索行业案例（确保流量贴合业务）
第二步规划：基于案例生成流量（核心目标落地）
第三步规划：对生成流量进行质量打分（可重试）
第四步规划：对生成结果做身份校验（满足业务质量要求）
```

该流程替代了 Agent 的"动态思考决策"，但本质仍是"目标 → 步骤拆解 → 执行"的 Agent 规划逻辑。

### 4.1.4 状态感知与记忆（State Awareness & Memory）

| 记忆类型 | 实现方式 | 存储内容 |
|----------|----------|----------|
| **短期记忆** | FastAPI 接口日志 | 行业/场景参数 → RAG检索结果 → 生成的流量 → 质量打分 → 身份判断结果 |
| **长期记忆** | Milvus 向量库 | 行业案例，可重复用于不同场景的流量生成 |

### 4.1.5 异常处理与自适应（Error Handling & Adaptation）

Agent 的"自适应能力"体现为"应对执行异常不中断"：

| 异常场景 | 处理策略 |
|----------|----------|
| MilvusRAGTool 向量库连接失败 | 降级策略：返回通用行业案例，而非直接报错 |
| TrafficIdentityTool 规则打分失败 | 兜底逻辑：直接调用 LLM 做身份判断 |
| LLM 调用超时/失败 | 重试机制 + 超时控制 |

| Checkpoints | 状态快照 | `checkpoints.db` |

#### 6.1.2 数据库 Schema
```sql
-- checkpoints.db 表结构 (由 langgraph-checkpoint-sqlite 自动管理)
-- 主要包含: thread_id, checkpoint_data, created_at

-- memory_store.db 表结构 (自定义)
CREATE TABLE IF NOT EXISTS memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    thread_id TEXT NOT NULL,
    content TEXT NOT NULL,
    embedding BLOB,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 6.2 数据流
```
用户输入 → messages (内存) → checkpoint (SQLite) → 下次对话恢复
                                    ↓
                            Milvus Lite (向量) ← RAG 检索 (权重中等)
```

### 6.3 流式输出
| 场景 | 用途 |
|------|------|
| LLM token 生成 | 实时显示 Agent 思考过程 |
| 工具调用 | 显示 "正在调用 xxx 工具..." |
| 检索进度 | 显示 "正在检索知识库..." |
| 最终结果 | 完整响应返回 |

```python
# 流式输出示例
@app.post("/api/chat/stream")
async def chat_stream(request: ChatRequest):
    async def generate():
        async for event in app.astream_events(input_data, config):
            if event["event"] == "on_chat_model_stream":
                yield f"data: {event['data']['chunk'].content}\n\n"
            elif event["event"] == "on_tool_start":
                yield f"data: [工具] 正在调用 {event['name']}...\n\n"
    return StreamingResponse(generate(), media_type="text/event-stream")
```

## 附录 A: 面试总结话术

> 以下话术可用于面试时描述本项目的 Agent 特征：

"我基于 LangChain+FastAPI 重构的流量生成工具，本质是一个结构化任务型智能体（Structured Task-oriented Agent），核心匹配 Agent 的五大特征：

**第一，目标驱动**：输入行业/场景后，自动完成'检索案例 → 生成流量 → 身份校验'全流程，最终输出带身份标签的模拟流量，无需人工干预；

**第二，工具调用**：封装了 RAG 检索、身份判断两个标准化 Tool，分别调用 Milvus 向量库和规则引擎，扩展 LLM 的行业数据能力和校验能力；

**第三，分步规划**：通过 LangChain 的 StateGraph 实现固定流程规划，将核心目标拆解为'检索-生成-校验'三步，按顺序执行；

**第四，状态记忆**：Milvus 存储的行业案例是长期记忆，接口请求日志记录的中间结果是短期记忆，支持流量生成的可追溯；

**第五，异常自适应**：针对向量库连接失败、LLM 调用超时等异常，设计了降级和重试策略，确保任务不中断。

该设计既保留了 Agent 的核心特征，又通过 SequentialChain 简化了动态决策逻辑，适配工业场景下'稳定优先'的需求，是典型的落地级 Agent 应用。"
